## 对软件赛题的直接启发

1. 不要做“会聊天”的 Agent，要做“能把事做完”的 Agent。
   评委会看现场 Demo 完成度、技术创新性、代码质量。一个能展示状态机、工具调用、失败恢复、任务日志的 Agent，比一个包装得很美的聊天窗口更有竞争力。

2. 48 小时里，选题要小而闭环。
   最适合的是一个 3 分钟能讲清楚、现场能跑通、失败时可降级的工作流：输入明确，过程可视，输出可验收。不要把场景摊太大。

3. 最值得押注的几个方向：
   - Long Horizon Agent：持续推进长任务，有记忆、状态、检查、回滚、复盘。
   - Ghost in the Workflow：替代重复劳动，例如表格核对、资料整理、会议纪要、项目追踪、自动报表。
   - Vertical Expert Agent：选一个垂直行业，做“懂流程、懂术语、能给出可执行结果”的专家 Agent。
   - Toolsmith：开发者生产力工具，例如自动测试、代码审查、README/文档生成、Issue 到 PR 的执行链路。

4. Demo 要按评分表倒推。
   现场必须表现出“真跑通”：真实输入、实时执行、可见中间状态、可复现结果。代码仓库要从第一天就持续提交，README 写清楚安装、运行、架构、限制和 Demo 流程。

5. 商业潜力要提前写进产品叙事。
   评分里创业潜力占 25%。项目一页纸需要明确：谁痛、为什么现在痛、当前替代方案是什么、你的 Agent 省了多少时间/减少多少错误、赛后怎么继续做。

6. 传播分不要临时补。
   2026-07-11 22:00 前至少发 2 篇过程记录帖是加分项。可以把 Build in Public 当成项目证据链：问题发现、架构草图、第一次跑通、失败修复、最终 Demo。

## 可落地选题方向

- 项目作战室 Agent：自动读取队伍 GitHub 提交、任务看板、聊天记录，生成 CP 提交材料、项目一页纸、Demo 脚本和小红书过程帖。
- 长线调研执行 Agent：给一个主题，自动检索、读网页/文档、整理证据、生成引用报告，并能在失败后重试和记录来源。
- 表格/飞书流程 Agent：把报名、设备需求、报销票据、团队信息等多表数据自动核对、补全、生成异常清单。
- 垂直专家 Agent：例如法律合同审查、医疗随访摘要、金融研报核验、工业巡检工单分析、教育错题诊断，重点展示行业知识、流程闭环和可追溯证据。
- 多模态会议 Agent：实时听会、读屏幕、抓白板/幻灯片，生成行动项、风险点和后续任务。
- 开发者 PR Agent：从 Issue 到代码修改、测试、README 更新、变更说明，展示工具调用链和回滚机制。
