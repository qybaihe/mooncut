## 产品是什么

MoonCut 是一个面向口播创作者的网页工具，完整流程是：

```text
聊想法 → 生成完整口播稿 → 提词录制 → 一键进入剪辑 → 等待处理 → 获得成片
```

产品只有两个主入口：

1. 剪辑页面
2. 录制页面 / 口播助手

不需要单独做首页、登录页、价格页或后台管理页。

## 必须实现的全局体验

- 桌面端和移动端都必须是完整设计，不能只是桌面布局缩小。
- 桌面端需要清晰的两个主入口；移动端需要适合单手操作的页面切换方式。
- 两个页面切换后必须保留各自状态。剪辑进度、上传的视频、聊天记录和口播稿不能因为切页丢失。
- 页面应当像可以正式发布的创作工具，而不是组件库拼装、后台管理模板或普通 AI Chat 页面。
- 所有主要操作必须真的可以点击并看到状态变化。
- 没有后端时使用可信的模拟流程，但界面中要让用户清楚这是本地演示。
- 中文文案要自然、克制、容易理解。
- 需要基本无障碍支持：键盘操作、清晰焦点、足够对比度、按钮触控区域不小于 44px。

## 页面一：剪辑页面

用户上传一条未经处理的口播视频，等待智能处理完成，最后得到新的口播成片。

必须覆盖以下状态。

### 1. 未上传

- 支持点击选择视频。
- 支持拖拽上传。
- 接受 MP4、MOV、WebM 等常见视频格式。
- 明确说明建议时长和本地演示状态。
- 简洁呈现主要能力，例如自动删除停顿、清理重复表达、生成节奏字幕。
- 上传入口必须是页面最明确的主操作。

### 2. 已上传，等待剪辑

- 使用 `URL.createObjectURL` 预览用户选择的真实本地视频。
- 显示文件名、文件大小和就绪状态。
- 可以更换视频。
- 提供少量有意义的剪辑选项，例如字幕风格和节奏强度，不要做成复杂专业剪辑器。
- 主操作为“开始智能剪辑”。

### 3. 处理中

- 使用前端计时器模拟完整处理过程。
- 不要只有一个转圈，需要表现清晰的处理阶段，例如：
  1. 读取口播内容
  2. 识别停顿与重复
  3. 整理字幕节奏
  4. 合成成片
- 展示进度百分比和当前阶段。
- 用户切换到录制页面再返回时，任务仍继续运行。

### 4. 处理完成

- 显示成片预览。
- 可以在“原片 / 成片”之间切换比较。
- 成片状态要有字幕、重点词或包装效果的前端视觉模拟。
- 展示处理结果数据，例如删除的停顿数量、精简的时长和重点词数量。
- 提供“下载成片”“重新剪一版”“换个视频”。
- 当前没有真实成片文件时，下载操作使用明确的演示反馈，不要伪造真实下载成功。

## 页面二：录制页面 / 口播助手

这个页面帮助用户从一个模糊想法得到完整口播稿，随后直接进入提词录制。

必须包含以下流程。

### 1. 与助手聊想法

- 助手首先询问用户最想让观众记住什么。
- 输入框支持 Enter 发送、Shift + Enter 换行，并兼容中文输入法。
- 提供快捷开场，例如：
  - 讲一个知识点
  - 做产品介绍
  - 分享个人经历
  - 表达一个观点
- 用户发送内容后，模拟助手思考和回复。
- 助手回复后提供 3 个可选择的内容建议，例如开头钩子、具体场景、结尾动作。
- 建议支持选择和取消选择。
- 用户可以把选中的建议加入完整口播稿。
- 需要有“换一组建议”的交互反馈。

### 2. 完整口播稿

- 展示一篇可以直接照着念的完整中文口播稿。
- 稿件必须可以直接编辑。
- 实时显示字数和预计口播时长。
- 提供几个简单的前端模拟润色操作：
  - 更口语
  - 再精简
  - 更有感染力
- 稿件自动保存到 `localStorage`。
- 可以复制稿件。
- 主操作为“进入提词录制”。
- 移动端需要在助手对话和完整稿件之间清晰切换，不能把两栏硬挤在一个屏幕里。

### 3. 提词录制

- 进入后切换到沉浸式录制界面，避免全局导航遮挡或误触。
- 优先使用浏览器 `getUserMedia` 获取摄像头与麦克风。
- 浏览器支持时使用 `MediaRecorder` 在本地真实录制。
- 相机权限拒绝、获取超时或浏览器不支持时，自动进入可继续体验的演示镜头，不得卡死。
- 提供：
  - 字号调节
  - 提词速度调节
  - 镜像开关
  - 3 秒倒计时
  - 开始、暂停、继续和完成录制
- 提词稿在镜头画面上清晰显示，当前句突出，已经念过的句子弱化。
- 录制中显示红点和计时。

### 4. 录制完成

- 浏览器实际录制成功时预览本地录制结果。
- 演示模式时显示明确的演示预览，不要声称已经生成真实文件。
- 提供：
  - 一键去剪辑
  - 重新录制
  - 保存 / 复制口播稿
- 点击“一键去剪辑”后自动切换到剪辑页面，并直接进入“已上传、等待剪辑”状态。
- 文件名可以显示为“刚刚录制的口播”。

## 实现约束

- 不接入真实后端，不调用收费 API。
- 不删除现有的可用功能来换取更简单的页面。
- 不要引入重量级 UI 组件库后直接套默认样式。
- 可以增加轻量依赖，但必须有明确价值。
- 本地视频和录制视频产生的 Blob URL 需要正确释放，避免内存泄漏。
- 摄像头、录制计时器、剪辑进度计时器在组件销毁时需要正确清理。
- 需要保留页面状态，但不能让已经被用户更换或删除的录制视频重新出现。
- 不需要实现真实视频剪辑；成片包装使用前端视觉模拟即可。

## 设计自由度

请重新建立完整的视觉方向，不需要沿用现有的暖白、黑色和绿色方案，也不要被当前组件结构限制。可以重写整体布局、设计语言、动效、导航、视频工作区、聊天方式和提词器表现。

最终结果需要具备明确的产品性格、强视觉记忆点和专业创作工具的完成度，同时保持任务路径直观。避免：

- 普通 SaaS 后台模板
- 大量同质化白色卡片
- 默认 Tailwind / 组件库质感
- 常见 AI 紫色渐变加发光球
- 只有装饰、没有信息层级的视觉效果
- 过大的空白 Hero 导致核心功能不在首屏
- 移动端导航遮挡输入框、视频控件或录制按钮
